## 1. 产品概述

Orvex 是一款面向开发者与效率用户的 Windows 桌面快速启动器，基于 Tauri 2.x + Rust 构建，具备极低内存占用与快速响应能力。用户可将常用应用、文件、网址分类管理，通过全局热键一键呼出面板，快速定位并启动目标。

**核心设计原则**

- **克制简约**：界面无冗余元素，操作路径最短
- **高级感**：视觉设计精致，支持自定义主题与背景
- **低打扰**：常驻后台，用完即隐，不占用工作区
- **开发者友好**：深度集成 Everything 搜索，支持 CRON 待办任务

## 4. 核心功能详述

### 4.1 快捷方式管理

#### 支持的项目类型
- 本地应用（.exe / .lnk 快捷方式）
- 本地文件 / 文件夹
- 网页 URL

#### 添加方式
- 拖拽文件 / 快捷方式到面板
- 右键菜单手动新增
- URL 输入框添加网页链接

#### 管理操作
- 右键图标：编辑名称、更换图标、移动分类、删除
- 左键长按拖拽：调整图标在当前分类中的排列顺序
- 以管理员身份运行：支持对特定应用提权启动
- 打开文件位置：在资源管理器中定位源文件

#### 权限边界：普通启动 vs 管理员身份运行

| 维度 | 普通启动 | 管理员身份运行 |
|---|---|---|
| 实现方式 | `std::process::Command` 直接 spawn | `ShellExecute` + `runas` 动词 |
| UAC 弹窗 | 无 | 有，用户需手动确认 |
| 适用场景 | 绝大多数日常应用 | 需要写注册表、系统目录的工具 |
| Orvex 自身权限要求 | 无需管理员权限运行 | Orvex 本身不需要提权，仅目标进程提权 |
| 失败处理 | 目标不存在时提示"文件未找到" | 用户取消 UAC 时，在面板底部短暂显示轻提示"已取消管理员启动"（Toast 样式，2 秒自动消失），不弹对话框、不报错 |
| 不支持场景 | — | 部分 UWP / Store 应用不支持 `runas`，右键菜单中对此类应用隐藏该选项 |

> **注意**：Orvex 不建议以管理员身份启动自身（会导致拖拽添加快捷方式失效），启动时检测到提权运行则在界面顶部显示警告横幅。

### 4.3 搜索功能

#### 内置搜索
- **快捷键模式**：呼出面板后输入关键词直接筛选当前分类
- **按键即搜模式**：任意字符触发搜索

#### Everything 集成搜索（可选插件）
- 需本地安装 Everything 1.4.x
- 搜索结果同时显示 Orvex 内置结果与 Everything 全盘结果
- 显示各来源结果数量统计

#### 搜索结果排序规则

混排时按以下优先级依次排列：

1. **Orvex 内置快捷方式**（精确名称匹配 > 前缀匹配 > 模糊匹配），同级内按使用频次降序
2. **Everything 全盘结果**，按 Everything 原生评分排序（文件名匹配度优先，路径深度次之）

Orvex 内置结果始终整体优先于 Everything 结果，不做交叉混排，两组之间以分隔线视觉区分。

**Orvex 内置评分权重**

| 匹配类型 | 权重 |
|---|---|
| 名称完全相等 | 100 |
| 名称以关键词开头 | 80 |
| 名称包含关键词 | 60 |
| 拼音首字母匹配 | 40 |

权重相同时，以 `launch_count`（启动次数）降序作为二级排序，`last_launch_at` 作为三级排序。

### 4.4 待办任务

#### 新建待办字段
- 待办名称（必填）
- 待办详情（选填）
- 运行方式：指定时间 / CRON 表达式
- 完成时间记录

#### 提醒方式
- 到达指定时间时弹出系统通知（Windows Toast）
- CRON 表达式支持周期性提醒

#### CRON 能力规格

**时区**：固定使用本机系统时区，不支持跨时区配置。时区信息在任务创建时由 Rust 后端读取 `Local::now()` 记录，不随系统时区后续变更自动迁移（变更时区后已有 CRON 任务需用户手动核查）。

**表达式格式**：采用 5 段标准 CRON（`分 时 日 月 周`），使用 `cron` crate 解析，不支持秒级精度。示例：

```
# 每天早上 9 点
0 9 * * *

# 每周一、三、五 18:30
30 18 * * 1,3,5

# 每月 1 日凌晨 0 点
0 0 1 * *
```

**表达式校验**：在用户输入完成失焦时实时校验，非法表达式给出明确错误提示（如"日期字段超出范围"），禁止保存；合法表达式实时预览下次触发时间。

**系统休眠 / 关机后的补偿策略**：
- Orvex 唤醒后以当前时间重新扫描所有待办任务，不回放休眠期间的每一个时间点
- 若指定时间任务（非 CRON）在休眠期间错过触发，且当前状态仍为 `pending`：唤醒后立即补触发一次，并在通知中注明"已延迟提醒"；补触发后立刻将任务状态置为 `missed`，同时写入 `compensated = true`，防止重复补偿
- 若用户在这次延迟提醒通知中点击"完成"，任务从 `missed` 转为 `done`
- CRON 周期任务在休眠期间错过的轮次**不补偿**，直接等待下一个周期；通知仅表示"本次提醒已到达"，不承担结束任务的语义

#### 待办状态机

```
         创建
          │
          ▼
       pending ──────────────────────────────► done
          │          用户手动标记完成 /              │
          │          到时触发后用户点击通知确认         │
          │                                         │
          │ 仅 datetime 类型：                       │
          │ 唤醒时发现触发时间已过，补触发一次提醒       │
          ▼                                         │
       missed ──────────────────────────────► done  │
                 用户在延迟提醒/历史列表中标记完成      │
```

**状态转换规则（定死）**：

| 状态 | 含义 | 进入条件 | 出现在哪个列表 |
|---|---|---|---|
| `pending` | 等待触发 | 任务创建时默认 | 待办列表 |
| `done` | 已完成 | **仅 datetime 类型**：用户手动标记完成；或任务触发后用户点击通知内"完成"按钮 | 历史待办 |
| `missed` | 已错过，未完成 | **仅 datetime 类型**：Orvex 唤醒时检测到触发时间已过，立即补触发一次延迟提醒，然后将状态置为 `missed` | 历史待办 |

**CRON 任务的任务级状态始终保持 `pending`**：CRON 是长期有效的提醒规则，不存在"完成整个任务一次就结束"的语义。每次触发只代表一次提醒事件，用户点击通知上的"知道了"仅关闭本次提醒，不修改 `status` / `done_at`。

**CRON 任务的结束方式**：若用户不再需要该周期提醒，直接删除该任务；CRON 任务不进入历史待办。

**`missed` 任务的后续处理**：进入历史待办，用户可手动标记为 `done`，或直接删除。不支持"重新激活"为 `pending`（避免状态回流导致重复提醒）。

#### 待办列表
- 待办列表：显示当前所有未完成任务
- 历史待办：显示已完成与已错过任务记录（即 `done` + `missed`）

#### 待办热键交互定义

按下 `hotkey.todo`（默认 `Ctrl+Shift+Q`）的行为：

- 若待办面板**未打开**：直接打开待办面板，默认停在"待办列表" tab，**不**自动触发新建
- 若待办面板**已打开且在前台**：关闭面板（toggle 行为）
- 若待办面板**已打开但被其他窗口遮挡**：将面板提到前台，不新建

> 热键不负责"直接新建"，新建走面板内右下角的"新建待办"按钮。这样热键语义单一（呼出 / 收起），不产生误操作。

## 5. 数据与存储

### 5.1 数据文件结构

所有数据以 JSON 格式存储于应用根目录下：

```
orvex/
├── data/
│   ├── app_data.json   # 快捷方式分类与项目数据
│   ├── config.json     # 用户配置（热键、外观、插件开关等）
│   └── todo.json       # 待办任务数据
├── icons/              # 自定义图标目录，文件名为 UUID，格式 PNG/ICO
│   ├── a1b2c3d4.png
│   └── ...
└── bak/
    └── ...             # 自动备份，保留最近 N 天（默认 7 天，可配置）
```

> **图标存储策略**：自定义图标不内嵌 base64 到 JSON，而是写入 `icons/` 目录，JSON 中仅存文件名（`icon:<uuid>.png`）。优点：主数据文件保持轻量，备份、diff、损坏恢复均不受图标体积影响；缺点：需在删除项目时同步清理孤立图标文件（启动时做一次 GC 扫描）。

### 5.2 数据模型

#### app_data.json

### 5.3 备份策略

**原子写入（防崩溃损坏）**

所有 JSON 文件的写入必须走以下原子流程，禁止直接覆盖目标文件：

```
1. 将新内容写入同目录临时文件（如 app_data.json.tmp）
2. 调用 flush() + fsync() 确保数据落盘
3. 调用 rename() 原子替换目标文件
4. 若任意步骤失败，删除临时文件，保留原文件不变
```

Rust 实现参考：`std::fs::rename()`在 Windows 上为原子操作（同卷），配合 `File::sync_all()` 使用。

**定期备份**

- 每次 Orvex 启动时触发一次备份（复制当前 `data/` + `icons/` 到 `bak/YYYY-MM-DD/`，形成可整体恢复的快照）
- 同一天多次启动只保留最后一次备份（覆盖当天目录）
- 保留最近 N 天的备份目录，N 默认 7，可在 `config.json` 中通过 `bak_days` 修改
- 数据文件损坏（JSON 解析失败）时，自动从 `bak/` 中找最近一份有效快照恢复；恢复内容必须同时包含匹配版本的 `data/` 与 `icons/`，避免 JSON 与图标引用不一致

**手动备份**

- 支持在设置页手动触发，将 `data/` + `icons/` 打包为 zip 导出
- 支持从 zip 恢复（导入前校验 JSON 格式，失败则整体拒绝，不做部分导入）

## 8. 多显示器与 HiDPI 验收标准

### 8.1 场景覆盖

| 场景 | 描述 |
|---|---|
| 单屏 100% | 标准 1080p，缩放比例 100% |
| 单屏 HiDPI | 2K / 4K，缩放比例 125% / 150% / 200% |
| 双屏同缩放 | 两块屏幕缩放比例相同 |
| 双屏异缩放 | 主屏 100%，副屏 150%（最常见的混合场景） |
| 跨屏移动 | 面板从主屏拖动至副屏 |

### 8.2 验收标准

**面板定位**
- 鼠标跟随模式下，面板呼出位置与鼠标物理坐标一致，误差 < 4px（以目标屏幕的逻辑像素计）
- 贴边隐藏在任意屏幕边缘均可触发，不因 DPI 偏移导致判定区域错位

**渲染清晰度**
- 应用图标在 200% 缩放屏幕上无明显锯齿，优先使用 256×256 或 SVG 图标源
- 文字、边框在所有缩放比例下无模糊（WebView2 的 `devicePixelRatio` 需随屏幕 DPI 动态更新）

**拾色器**
- 跨屏取色时返回的 HEX 值与目标像素实际颜色一致，允许误差 ±1（8bit 每通道）
- 异缩放双屏下取色坐标换算正确，不出现坐标偏移导致取到错误位置的颜色

**窗口迁移**
- 面板从低 DPI 屏拖至高 DPI 屏后，内容自动以目标屏 DPI 重绘，不出现内容缩放错误
- 窗口位置记忆（若有）存储逻辑坐标而非物理坐标，重启后恢复位置正确

**性能**
- 异缩放双屏环境下，热键呼出响应仍满足 < 100ms 的要求

## 9. 非功能性需求

### 9.1 性能指标

| 指标 | 目标 |
|---|---|
| 冷启动时间 | < 1 秒 |
| 热键呼出响应 | < 100ms |
| 常驻内存占用 | < 50MB |
| 安装包体积 | < 15MB |

### 9.2 稳定性

- 数据文件损坏时自动降级至最近备份
- 热键注册失败时给出明确提示，不静默失败
- Everything 未安装时对应功能优雅降级，不影响其他模块

### 9.3 可维护性

- Rust 后端按模块拆分：`hotkey` / `tray` / `launcher` / `storage` / `todo` / `search`
- 前端按功能拆分 React 组件，样式统一使用 Tailwind
- Tauri Command 接口文档化，前后端边界清晰
